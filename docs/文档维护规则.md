## AI助手自动化工作流

### ⚡ 核心原则：主动执行，无需手动提醒

**重要说明：** 本章节面向AI助手（如Claude Code），明确在版本发布时应**自动执行**的文档更新流程，无需用户手动要求。

---

### 触发条件：何时自动执行文档更新

当用户的对话中出现以下关键词或场景时，AI助手应**立即主动**执行文档更新检查：

#### 1. 明确的版本发布指令
```
用户说：
- "发布新版本 v1.3.0"
- "准备发布 v1.3.0"
- "版本 v1.3.0 开发完成"
- "打包发布 v1.3.0"
- "我要发布新版本了"
```

#### 2. 功能开发完成的标志
```
用户说：
- "XXX功能已经开发完成"
- "这个功能做好了，可以发布了"
- "所有功能都完成了"
- "准备打包"
```

#### 3. Git提交/标签相关
```
用户说：
- "创建版本标签"
- "git tag v1.3.0"
- "准备提交所有代码"
- "要push到GitHub了"
```

#### 4. CI/CD构建相关
```
用户说：
- "触发CI/CD构建"
- "开始自动构建"
- "准备发布到生产环境"
```

---

### 自动执行流程

当检测到上述触发条件时，AI助手应**主动**执行以下步骤：

#### 第0步：确认版本信息（必须）

**立即询问用户以下信息（如果未明确）：**

```
我检测到您准备发布新版本，让我先确认一些信息：

1. 新版本号是什么？（如：v1.3.0）
2. 本次更新的主要内容是什么？
   - 新功能：[列出]
   - Bug修复：[列出]
   - 性能改进：[列出]
   - 其他变更：[列出]
3. 预计发布日期：[YYYY-MM-DD]

确认后，我将自动更新所有相关文档。
```

#### 第1步：自动执行完整文档更新

**无需用户再次提醒，直接执行：**

使用TodoWrite工具创建任务列表：
```json
[
  {"content": "更新版本历史.md", "status": "in_progress"},
  {"content": "更新已完成任务.md", "status": "pending"},
  {"content": "更新项目路线图.md", "status": "pending"},
  {"content": "更新主README.md", "status": "pending"},
  {"content": "更新相关功能文档", "status": "pending"},
  {"content": "更新所有文档日期标记", "status": "pending"}
]
```

按照[版本更新流程](#版本更新流程)章节的6个步骤依次执行。

#### 第2步：执行质量检查

**自动运行检查清单：**

使用[发布新版本时的检查清单](#发布新版本时的检查清单)验证：
- ✅ 所有必须更新的文档已更新
- ✅ 版本号在所有文档中一致
- ✅ 所有文档的"更新时间"已更新
- ✅ 无明显格式错误

#### 第3步：总结并报告

**完成后，向用户提供清晰的总结报告：**

```markdown
✅ 版本 v1.3.0 文档更新完成

## 已更新的文档：
1. ✅ docs/05-项目管理/版本历史.md

6. ✅ docs/01-快速开始/常见问题FAQ.md

## 版本信息：
- 版本号：v1.3.0
- 发布日期：2025-11-15
- 完成度：98% → 100%

## 主要变更：
- ✨ 新功能：未来五年牛头数预测
- 🐛 Bug修复：近交系数计算精度
- 🔧 改进：Excel报告生成速度提升30%

## 下一步建议：

### 特殊情况处理

#### 情况1：用户只是询问，并非真要发布

**用户说：**
```
"如果我发布v1.3.0，需要做什么？"
"下次发布新版本时怎么办？"
```

**AI助手应该：**
- ❌ 不要立即执行文档更新
- ✅ 回答流程说明，提供指引
- ✅ 询问："您现在需要我帮您发布新版本并更新文档吗？"

#### 情况2：小版本号更新（Bug修复版本）

**用户说：**
```
"发布 v1.2.0.15，修复了一个小Bug"
```

**AI助手应该：**
- ✅ 执行简化流程（仅更新版本历史和README版本号）
- ✅ 告知用户："这是Bug修复版本，我已更新版本历史和主README"

#### 情况3：用户明确说"不需要更新文档"

**用户说：**
```
"发布v1.3.0，但文档先不更新"
```

**AI助手应该：**
- ✅ 尊重用户意愿，不执行文档更新
- ⚠️ 提醒用户："好的，我不会更新文档。但请记得稍后更新文档，以保持文档与代码的同步。"

---

### 自动化最佳实践

#### ✅ 做到这些：

1. **主动识别触发条件**
   - 分析用户对话内容
   - 识别版本发布相关关键词
   - 主动询问是否需要更新文档

2. **一次性完成所有更新**
   - 不要分多次询问
   - 一次性完成所有必要的文档更新
   - 使用TodoWrite跟踪进度

3. **清晰的进度反馈**
   - 告诉用户正在执行的步骤
   - 实时更新TodoWrite状态
   - 完成后提供详细总结

4. **保持灵活性**
   - 根据实际情况调整流程
   - 如果缺少信息，主动询问用户
   - 特殊情况特殊处理

#### ❌ 避免这些：

1. **等待用户明确要求**
   - ❌ "您需要我更新文档吗？"（应该主动执行）
   - ✅ "我现在开始更新所有相关文档..."

2. **遗漏文档更新**
   - 使用检查清单确保完整性
   - 按照标准流程执行

3. **过度询问**
   - 一次性收集所有必要信息
   - 不要重复询问相同的问题

## 版本更新流程

### 发布新版本时的文档更新步骤

每次发布新版本（如 v1.2.0.14 → v1.3.0）时，按以下顺序更新文档：

#### 第1步：更新版本历史
**文件:** `docs/05-项目管理/版本历史.md`

1. 在文件顶部添加新版本条目
2. 按类别记录所有变更：
   - ✨ 新功能
   - 🐛 Bug修复
   - 🔧 改进
   - 📚 文档更新
3. 包含发布日期和版本号

**示例:**
```markdown
## [v1.3.0] - 2025-11-15

### ✨ 新功能
- 未来五年牛头数预测
- 选配方案对比预测

### 🐛 Bug修复
- 修复近交系数计算精度问题

### 🔧 改进
- 优化Excel报告生成速度（提升30%）

### 📚 文档
- 更新快速入门指南
- 新增预测功能使用说明
```

#### 第2步：更新已完成任务清单
**文件:** `docs/05-项目管理/已完成任务.md`

1. 将本版本完成的功能添加到相应分类
2. 更新完成度百分比
3. 更新"最后更新"日期

**需要更新的位置:**
- 文件开头的版本号和更新时间
- 对应功能模块的任务清单
- 总体完成度统计

#### 第3步：更新项目路线图
**文件:** `docs/05-项目管理/项目路线图.md`

1. 将已完成的高优先级任务移除或标记为完成
2. 调整剩余任务的优先级
3. 添加新发现的需求或任务
4. 更新"当前版本"和"最后更新"时间

#### 第4步：更新主README
**文件:** `docs/README.md`

**必须更新的位置:**
```markdown
**当前版本:** v1.3.0        ← 更新版本号
**更新时间:** 2025-11-15     ← 更新日期
**完成度:** 98%              ← 更新完成度

## 归档管理

### 何时归档文档

**归档时机:**
1. 文档内容已被整合到新文档中
2. 功能已废弃或被新功能替代
3. 文档内容过时且不再适用

### 归档步骤

**步骤1：创建归档说明**

在归档文件顶部添加归档声明：
```markdown
# [原文档标题]

> **⚠️ 归档说明**
>
> 本文档已于 2025-11-03 归档。
>
> **归档原因:** 内容已整合到新文档
>
> **替代文档:** [新文档标题](../相对路径/新文档.md)
>
> ---
>
> 以下是归档内容：
```

**步骤2：移动到归档文件夹**

```bash
# 移动到对应的archive文件夹
mv docs/功能文档/旧文档.md docs/功能文档/archive/旧文档.md
```

**步骤3：更新archive/README.md**

在归档文件夹的README.md中添加条目：
```markdown
| 归档文件 | 归档日期 | 归档原因 | 替代文档 |
|---------|---------|---------|---------|
| 旧文档.md | 2025-11-03 | 内容已整合 | [新文档](../新文档.md) |
```

### 归档文件夹结构

```
docs/
├── 功能文档/
│   └── archive/
│       ├── README.md
│       └── [归档的文档]
├── 02-用户手册/
│   ├── 01-数据管理/
│   │   └── archive/
│   ├── 02-育种计算/
│   │   └── archive/

```

### 归档保留期限

- **短期归档**（1-3个月）：最近整合的文档，用于对照参考
- **长期归档**（> 3个月）：历史文档，保留用于追溯
- **永久保留**：重要的历史设计文档、关键决策记录

**清理规则:**
- 归档超过6个月且无价值的文档可以删除
- 删除前确认无其他文档引用
- 重要历史文档永久保留

## 命名和格式规范

### 文件命名规则

**文件夹命名:**
```
✅ 正确：01-快速开始/
✅ 正确：02-用户手册/
❌ 错误：快速开始/  （缺少数字前缀）
❌ 错误：1-快速开始/（数字位数不足）
```

**文档命名:**
```
✅ 正确：快速入门指南.md
✅ 正确：Excel报告生成指南.md
✅ 正确：API参考.md
❌ 错误：guide.md（使用英文且不够描述性）
❌ 错误：快速入门.txt（错误的文件扩展名）
```

**命名原则:**
- 使用中文描述性名称
- `.md` 扩展名
- 避免特殊字符（除了下划线和连字符）
- 文件夹使用数字前缀（01-05）确保排序

### 文档格式规范

**文档头部信息:**

每个文档应包含以下元信息（根据文档类型选择）：
```markdown
# 文档标题

**更新时间:** 2025-11-03
**适用版本:** v1.2.0.14+
**作者:** [可选]

---
```

**目录结构:**

复杂文档应包含目录：
```markdown
## 📋 目录

1. [章节一](#章节一)
2. [章节二](#章节二)
3. [章节三](#章节三)

---
```

**标题层级:**
```markdown
# H1 - 文档标题（每个文档只有一个）
## H2 - 主要章节
### H3 - 子章节
#### H4 - 细节说明
```

**代码块:**
```markdown
**示例:**
```python
# Python代码使用python标记
def example():
    pass
```\u0060

```bash
# Bash命令使用bash标记
npm install
```\u0060

```json
// JSON使用json标记
{
  "key": "value"
}
```\u0060
```

**强调标记:**
```markdown
✅ 正确/已完成
❌ 错误/未完成
⚠️ 警告/注意
💡 提示
📝 备注
⏰ 进行中
🔴 高优先级
🟡 中优先级
🟢 低优先级
```

**链接格式:**
```markdown
[链接文字](相对路径/文档.md)  ← 站内链接使用相对路径
[外部链接](https://example.com)  ← 外部链接使用完整URL
```

**表格格式:**
```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 值1 | 值2 | 值3 |
```

### Markdown最佳实践

**列表缩进:**
```markdown
- 一级列表
  - 二级列表（2个空格缩进）
    - 三级列表（4个空格缩进）
```

**空行使用:**
- 标题前后各一个空行
- 段落之间一个空行
- 代码块前后各一个空行

**中英文混排:**
- 中英文之间不需要空格（GitHub渲染会自动处理）
- 英文单词和标点之间需要空格

## ❓ 常见问题

### Q1: 如果忘记更新某个文档怎么办？

**A:** 在下一次更新时补充：
1. 在版本历史中添加补充说明
2. 更新遗漏的文档
3. 在维护记录中标注

### Q2: 如何处理文档冲突？

**A:** 如果多人同时编辑：
1. 使用Git分支管理文档变更
2. 发布前合并所有文档更新
3. 指定文档负责人协调冲突

### Q3: 是否需要为小版本（如v1.2.0.14 → v1.2.0.15）更新所有文档？

**A:** 取决于变更内容：
- **仅Bug修复**: 只更新版本历史和README版本号
- **小功能添加**: 更新相关功能文档
- **大功能添加**: 完整更新流程

### Q4: 如何确保文档质量？

**A:** 建立审核流程：
1. 文档编写者自查
2. 同行评审（如有团队）
3. 用户测试反馈
4. 定期质量检查
